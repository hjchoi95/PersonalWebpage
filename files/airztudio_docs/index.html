Style Guidelines

Planning and Coding Standard Guidelines

Planning standards:

  1. Before writing any new module or function, the corresponding class diagram must be updated. The class diagram shall be a UML class diagram.
  2. Proposed: In addition to the class diagram, a UML sequence diagram of all current routes must also be maintained.

Organizational standards:

  1. Files under models/ are reserved for functions that directly model some behavior in the server.
  2. Files under controllers/ are reserved for functions that interface between routes and models.
  3. Files under routes/ are reserved for functions that route requests to the appropriate render web page calls and manage session logic.

Coding standards:

  1. All functions and variable names must use camel casing.
  2. The documentation generator we are using is JSDoc. All functions must also be documented with the parameters, function, and return type specified using JSDoc syntax. The primary author of the function should also be documented. This must be written before the function is implemented.
  3. Modules should likewise be documented with a clear purpose.
  4. All algorithmic functions must include examples.
  5. Algorithmic functions are defined as functions whose purpose is solely to manipulate and format javascript data.
    • Examples of algorithmic functions:
      • date formatters
      • array sorting
      • sql query sanitizer
    • Examples of non-algorithmic functions:
      • view tables from a database
      • error handlers
  6. All incomplete functions must CLEARLY be marked as "stub" in the documentation.
  7. All operators, besides function-like operators and increment operators, must be surrounded by spaces (eg, a === b instead of a===b)
  8. Braces for a javascript function must start right after the declaration, not on the next line. They must also be separated by a space. eg. 
    function() {  
  //code  
}
    instead of: 
    function()  
{
  //code  
}
  9. Braces for non-function keywords (eg. if, switch, for, typeof) must also follow the convention outlined in point 6.
  10. For non-function keywords (eg. if, switch, for, typeof), the keyword and the parentheses and braces must be separated by a space. eg. 
    if (a == b) {  
  //code  
} else if (a == c) {  
  //code  
} else {  
  //code  
}
    instead of: 
    if(a==b){  
  //code   
}else if(a == c) {  
  //code   
} else{  
  //code  
}
  11. If statements that do not have parentheses are banned.
  12. When calling a function, the parentheses must always be right next to the function name or the function keyword. eg. 
    function() {  
  //code  
}
    instead of: 
    function () {  
  //code   
}
  13. The length of every line of javascript must not exceed 80 characters. Exceptions will be granted if exceeding 80 characters significantly increases readability, but this exception will be rare.
  14. When defining objects, every item must be separated by a newline. The brace must also be separated by a newline. All items must be indented as appropriate to its nesting level. Exception will be granted for a small object (less than 15 characters) being fed as a function argument. However, in this case, spaces must be added for the right side of colons, commas and next to braces. eg. 
    {  
  "status": "FAIL: authentication failed",  
  "result": null  
};
    or 
    fun1({ "abc": "def", "ghi": "jkl" });
    instead of 
    {"status":"FAIL: authentication failed","result":null}
  15. All statements must end with a semicolon.
  16. All tabs must be soft tabs ("spaces") of length 2. Avoid using hard (ie. "real") tabs.
  17. Every non-trivial function of a model must be tested. There must be at least two (ideally three) test cases for every domain the input is split into. A domain is defined as a contiguous set of values for which the output of the function is similar.
    • For example, an absolute value function has two domains, negative numbers and positive numbers. The test cases for the domains should be randomly in the middle of the domain, and at the boundaries of the domain.
  18. NEVER push code that breaks othersâ€™ code.
  19. All strings in javascript must use double quotes instead of the single quote. An exception will be made for strings that contain more than 4 double quotes and no single quotes, in which case the string can be single-quoted to avoid escaping.
  20. No magic numbers: please do not put random numbers in the middle of code unless the meaning is obvious. Instead, declare it a const variable at the top of the file give it a suitable name that clearly expresses intent.
